# Skill Internationalisation (Phase-C C21)

copilot-omni skills are shipped as Markdown with YAML-ish frontmatter. The
canonical language is English — `skills/<name>/SKILL.md` is the source of
truth for executable logic. Optional translations override the user-facing
fields (`description`, `argument-hint`, the `<Purpose>` body) without
duplicating the skill's bash/python blocks.

## Layout

```
skills/<name>/
  SKILL.md                   # English, authoritative
  translations/
    es.md                    # Spanish
    fr.md                    # French (example)
    ja.md                    # Japanese (example)
```

The translation file uses the same frontmatter keys as the base SKILL.md,
plus a `lang: <code>` field. Only the user-facing fields are consumed —
everything inside `\`\`\`bash` / `\`\`\`python` fences in the base SKILL.md
is run unchanged regardless of locale.

## Activation

Set `OMNI_SKILL_LANG` to the desired locale:

```bash
export OMNI_SKILL_LANG=es
python3 scripts/skill_i18n.py resolve omni-plan
```

When the env var is unset, equal to `en`, or missing a matching
translation, the English SKILL.md is returned unchanged. All fallbacks
are silent — localisation is an enhancement, never a failure mode.

## Adding a language

1. Copy the frontmatter from the base `SKILL.md` into
   `skills/<name>/translations/<lang>.md`.
2. Translate `description`, `argument-hint`, and the `<Purpose>` block.
3. Do NOT translate bash/python fences, tool names, slash-commands, or
   frontmatter keys — those are user-visible identifiers that must stay
   stable across locales.
4. Add `lang: <code>` to the translation frontmatter.
5. Commit. `scripts/skill_i18n.py list-translations <skill>` will now
   surface the new code.

## Shipped translations

| Skill | Languages |
|---|---|
| `plan` (`omni-plan`) | `es` (Phase-C C21 proof of concept) |

Contributions welcome — each new language adds a single `<lang>.md`
file; there is no central registry to update.

## Non-goals

- Run-time UI translation of tool output. Only skill descriptions are
  localised.
- Translation of `docs/*.md`. Those live in English to keep the main
  documentation set tractable.
